项目规划书：VSCode 扩展 “C++ 项目脚手架”
1. 项目概述
本扩展旨在为 C++ 开发者提供一个轻量、零配置的快速原型验证环境。用户只需编写 .cpp 文件，扩展将自动解析 #include 依赖，无需 CMakeLists.txt 即可完成编译和运行，并可一键导出 compile_commands.json 以激活 clangd 的完整智能提示。核心价值是“写即运行”，显著降低 C++ 碎片化代码测试的门槛。

2. 技术选型与架构
语言 & 运行时：TypeScript + Node.js (VSCode 扩展标准技术栈)

关键依赖：

@types/vscode：VSCode API 类型

typescript、webpack（打包）

fast-glob：快速文件查找

comment-json：解析带注释的 JSON（用于特殊注释库导入）

minimatch：路径 glob 匹配（用于 include 路径检索）

child_process：执行 clang/clang++ 命令

crypto：文件内容哈希（缓存）

架构设计：

Core Module：依赖图构建、编译命令生成、缓存管理

Config Module：工作区配置读取、用户/工作区设置合并

Command Module：命令注册（创建项目、编译、运行、导出 compile_commands）

File Watcher Module：监听文件变更，触发增量编译/依赖图更新

UI Module：状态栏按钮、快捷键、模板选择 QuickPick

3. 功能模块拆解与实现策略
功能模块	实现要点
项目模板创建	提供常见模板（Console, Header-only, Library）通过 QuickPick 选择，在指定目录生成 .vscode/cpp-scaffold.json 配置文件及示例 .cpp
依赖自动识别	使用正则表达式提取所有 #include "..." 和 #include <...>。对 "..." 先在当前目录查找，再根据配置的 include 路径查找。如果找到同名 .cpp，则标记为联合编译单元；否则视为外部库，只参与头文件搜索路径 -I。
单文件直接运行	若当前打开或激活的 .cpp 文件无联合编译的 .cpp 依赖（自包含），则直接生成 clang++ file.cpp -o file && ./file 命令并执行，输出到集成终端。
特殊注释导入库	解析 C++ 注释中的特殊标记，例如 // @lib: /path/to/libfoo.a 或 // @lib: -lcurl，自动转换为 -l 或直接路径添加到链接参数。支持静态库和动态库路径。
compile_commands.json 导出	在后台维护所有被识别为编译单元的文件列表及其编译命令（基于依赖图），通过命令导出标准格式 JSON，clangd 可直接加载。同时监听文件增删改，自动更新。
include 与库路径配置	使用 .vscode/cpp-scaffold.json 工作区配置，字段 includePaths, libraryPaths。#include 解析时按顺序在这些路径中查找头文件。
缓存优化	计算每个编译单元的依赖文件集合及其修改时间、头文件内容哈希，生成编译缓存键。当依赖未变时，跳过编译，直接使用已有产物。缓存信息存储在 .vscode/scaffold-cache.json。
4. 项目排期（总计 8 周，MVP 版本）
项目采用敏捷迭代，划分为五个阶段，每周工作 5 天（按单人全职估算）。

第一阶段：基础架构与项目创建（第1-2周）
任务：

初始化 VSCode 扩展项目，配置 TypeScript、Webpack 打包。

实现扩展激活事件、命令注册（cppScaffold.newProject、cppScaffold.runFile 等）。

设计工作区配置文件 JSON Schema，读写逻辑。

实现项目模板引擎：从模板文件夹复制并替换占位符。

开发 QuickPick 交互流程。

产出：可创建预置模板的项目，能生成基础配置文件。

第二阶段：核心依赖分析引擎（第3-4周）
任务：

实现 #include 正则提取（处理注释中 include、续行符等）。

实现头文件查找算法：先本地相对路径，再遍历配置的 include 路径。

构建文件依赖图（有向无环图），识别联合编译 .cpp 与纯头文件。

单元测试：覆盖各种 include 模式。

产出：给定一个入口 .cpp 可输出完整编译单元列表和 -I 路径。

第三阶段：编译运行与特殊注释解析（第5-6周）
任务：

生成 clang++ 编译命令（包括所有联合编译的 .cpp、-I 路径、特殊注释引入的 -l/库路径）。

实现特殊注释解析器，支持 // @lib: 和 /* @lib: ... */。

集成终端执行编译命令，显示输出，错误匹配并跳转。

实现单文件运行功能。

导出 compile_commands.json 功能（包含所有编译单元的命令）。

添加文件监听，依赖变化时自动重建并更新 compile_commands.json。

产出：可以右键或快捷键运行当前单文件，自动编译并执行，库注释正常工作，clangd 获得正确索引。

第四阶段：缓存优化与配置集成（第7周）
任务：

设计缓存数据结构：依赖文件签名（mtime + 大小 + 内容哈希可选）、编译选项签名。

缓存读写逻辑，编译前检查缓存，命中则跳过。

文件删除/新增时使相关缓存失效。

用户设置接口：cpp-scaffold.includePaths、cpp-scaffold.libraryPaths 等全局默认值，与工作区配置合并。

状态栏提示当前项目状态。

产出：二次编译几乎瞬间完成，用户可配置全局路径。

第五阶段：测试、文档与发布准备（第8周）
任务：

编写集成测试（模拟工作区各种结构）。

性能测试（大项目依赖图构建耗时）。

用户文档编写，包含 GIF 演示。

生成 Changelog，准备发布至 VSCode Marketplace。

修复内测反馈的 Bug。

产出：可发布的稳定版本 1.0.0。

5. 风险管理
编译器依赖：必须安装 clang/clang++ 且配置正确。需在激活时检测并给出友好提示。

复杂 include 模式：如 #include MACRO 或包含计算路径的情况，初始版本仅支持字面量字符串，后续可扩展。

Windows 兼容性：路径分隔符、shell 执行差异，优先支持 bash-like 终端（Git Bash 或 WSL），但基础功能需兼容 cmd。

性能：大项目下全量依赖图构建可能较慢，缓存机制可缓解。后续可考虑 Worker 线程异步构建。

6. 可扩展性设计
插件化编译工具链：目前硬编码 clang，预留接口支持 gcc/msvc。

模板市场：允许用户贡献 GitHub 仓库作为模板源。

更丰富的注释标记：如 // @flag: -std=c++20，支持自定义编译选项。

以上规划书和排期为初始版本设定，可按实际资源调整工期。优先交付 MVP 功能，快速收集用户反馈迭代。
